# Package scaffolding, TypedDict annotations, and exception hierarchy for Liu Yao engine

This change turns a loose collection of Python modules into a proper installable package (`pyproject.toml`), adds a custom exception hierarchy (`LiuyaoError` with three subtypes), introduces TypedDict annotations for the wangshuai and dongbian return values, wraps the analysis orchestrator steps in graceful-degradation try/except blocks, and lands ~116 new unit tests across three new test files. A CI workflow (GitHub Actions, Python 3.9-3.12) and a docs reorganization (moving 25 reference markdown files into `docs/reference/`) round out the housekeeping.

**Watch for:** The graceful-degradation pattern in `run_analysis` silently swallows failures and returns sentinel values (empty lists, `None`, hardcoded "neutral" dicts), which can mask bugs in production (confirmed). The dual-analysis path (`run_dual_analysis`) lacks the same protection and will crash on the same errors that `run_analysis` now absorbs (confirmed). The CI workflow installs `sxtwl` separately rather than relying on the declared dependency in `pyproject.toml`, indicating a possible packaging gap (confirmed).

## High-level view

The graceful-degradation strategy catches `LiuyaoError` around six analysis steps (jixiong, yingqi, guaci, shuanghe, tuopu, fushen, xintai) and substitutes inert defaults. This prevents a single sub-analysis failure from aborting the entire report, but creates an asymmetry: `run_analysis` degrades while `run_dual_analysis` propagates the same exceptions unguarded. Callers of the dual path get different failure semantics for the same underlying operations.

The exception hierarchy uses multiple inheritance (`ArrangementError(LiuyaoError, ValueError)`) to preserve backward compatibility with existing `except ValueError` clauses. The only gap is that `AnalysisError` is defined and exported but never raised anywhere in the diff.

The CI matrix covers Python 3.9-3.12 with a separate `pip install sxtwl` step after the editable install. This reveals that `sxtwl` as declared in `[project].dependencies` may not resolve correctly via the normal install path on all platforms.

## Graceful degradation in the analysis orchestrator

The orchestrator (`run_analysis`) wraps steps 4 through 9 in individual `try/except LiuyaoError` blocks. Each catch substitutes a specific sentinel: jixiong gets `{"pattern": "Õłåµ×ÉÕ╝éÕĖĖ", "ji_xiong": "Õ╣│", "explanation": "..."}`, yingqi gets `[]`, everything else gets `None` or a reset to `"normal"`.

The sentinel for jixiong is the most consequential. Downstream code (the report formatter, the dual-perspective consensus logic) reads `report.jixiong_result["ji_xiong"]` to determine the overall verdict. When an analysis exception occurs, the result silently becomes "Õ╣│" (neutral) rather than indicating uncertainty. A caller cannot distinguish "the hexagram genuinely reads neutral" from "the analysis crashed." A `partial: bool` flag or a `warnings: List[str]` field on `AnalysisReport` would make failure visible without crashing.

The dual-analysis path calls `judge_jixiong` and `analyze_yingqi` directly without protection:

```python
report.jixiong_result = judge_jixiong(
    hexagram, yong_shen, shared_ws, shared_db, question_type
)
report.yingqi_results = analyze_yingqi(
    hexagram, report.yong_shen_lines, shared_ws, shared_db
)
```

If `judge_jixiong` raises `LiuyaoError` for one perspective, the entire `run_dual_analysis` call fails. Either both paths should degrade or both should propagate.

Steps 2 (wangshuai) and 3 (dongbian) remain unprotected in `run_analysis`. If those crash, the report has no fallback. The boundary of "what degrades" is implicit rather than documented.

## Exception hierarchy -- unused AnalysisError

`AnalysisError` has zero raise sites in the codebase. The `except LiuyaoError` blocks in the orchestrator would catch it if raised from a sub-module, but no sub-module currently raises it. Exporting an unused exception class in `__all__` sets an API contract that isn't yet backed by implementation.

## CI workflow and dependency management

```yaml
- run: pip install -e '.[test]'
- run: pip install sxtwl
```

If `sxtwl` is already in `[project].dependencies`, the second line is redundant unless the package can't be resolved during the editable install (e.g., missing wheels for some Python versions). The fact that it's listed separately signals the dependency declaration may not work reliably across the matrix. End users doing `pip install zhushi-liuyao` would hit the same problem.

## Test coverage gaps

The three new test files total ~116 tests covering wangshuai states and overrides, dongbian predicates with positive/negative cases, and jixiong yong-shen selection plus judgment via real hexagrams.

Not tested: the orchestrator's degradation behavior (no test mocks a `LiuyaoError` mid-flow), the `CalendarError` wrapping (no test passes an invalid date through `get_gan_zhi`), the dual-analysis path under failure, and the guaci/shuanghe/xintai sub-modules.
